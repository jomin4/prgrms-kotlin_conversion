# step-01: 신규 프로젝트에는 코프링을 사용하는게 권장됩니다

- 강의 링크: https://www.slog.gg/p/14128#1강
- 상태: 완료

## 요구사항 요약

코드 타이핑 없이 개념만 이해하는 강의. 기존 프로젝트는 자바로 유지보수하되, 신규 프로젝트는 코틀린(코프링)을 권장하는 이유를 정리.

- Null 안정성: 코틀린은 `?`로 nullable 여부를 컴파일 타임에 강제 → 자바의 런타임 NPE를 사전에 차단
- 보일러플레이트 감소: `data class`로 게터/세터/`equals`/`hashCode`/`toString`을 롬복 없이 해결 (10~11강에서 롬복 제거 시 체감)
- Spring과의 궁합: `kotlin("plugin.spring")`이 `@Component`/`@Service` 등을 자동 `open` 처리 → 코틀린 기본 `final` 클래스와 스프링 프록시 AOP 충돌 방지 (7강에서 직접 다룸)
- 자바와 100% JVM 바이트코드 상호운용 → 파일 단위 점진적 전환 가능 (이 강의 전체의 전제)
- 기존 프로젝트를 그대로 두는 이유: 무리한 전면 전환의 리스크(러닝커브, 빌드 재검증, 롬복 제거 등)가 신규 설계 대비 이득보다 클 수 있음
