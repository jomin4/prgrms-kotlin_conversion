> 새 세션을 시작할 때 이 문서 전체를 첫 메시지로 붙여넣으면 동일한 방식으로 이어서 진행할 수 있다.

## 이번 강의 정보

- 강의명/링크: 코프링으로 전환 (https://www.slog.gg/p/14128) — 자바 스프링부트 프로젝트를 코틀린(코프링)으로 점진 전환하는 67강 시리즈
- 저장소 폴더: `kotlin_conversion` (GitHub: https://github.com/jomin4/prgrms-kotlin_conversion, public)
- 기술 스택: Spring Boot 3.5.4 + Kotlin(Gradle Kotlin DSL), JPA/H2, Spring Security + OAuth2(카카오/구글/네이버 소셜로그인) + JWT, Next.js/TypeScript 프론트엔드
- 베이스 코드: 강사의 자바 원본(`jhs512/p-14184-1`)을 clone 후 `.git` 제거, 새 레포로 재초기화한 상태 (Member/Post/PostComment 게시판 API)
- 단계 번호 매핑: **강의의 "N강"을 그대로 "단계 번호"로 사용** (1강 = step-01, 67강 = step-67)
- 진행 방식: 매 강마다 원본 강의 자료(slog.gg 페이지)의 요구사항과 강사의 실제 커밋을 참고해 코드와 설명을 제공한다. **(10강까지는 사용자가 직접 타이핑, 10강부터는 사용자가 승인하면 에이전트가 직접 파일에 반영하는 방식으로 전환)**

## 역할 분담

- **사용자**: 에이전트가 제시한 코드/설명을 검토하고 **승인**한다 (예: "좋아", "진행해", "승인", 특별한 문구 강요 없이 자연스러운 승인 표현이면 됨). 1~9강은 사용자가 직접 타이핑했으나, **10강부터는 승인만 하면 에이전트가 파일에 직접 반영**한다.
- **에이전트**: 환경 세팅, 단계별 코드·설명 제공, (10강부터는) 소스 코드 파일 직접 반영, 빌드 검증, git 커밋/푸시 자동화, 학습 로그 기록을 전부 담당한다.

## 진행 루틴 (매 단계 반복)

1. 해당 강의 요구사항과 코드를 제공한다 (개념 위주 강의는 코드 없이 설명만 제공될 수 있음).
   - **코드 제시 형식 (10강부터 확정)**: 자바 원본과 코틀린 결과물을 전체 다 나란히 보여주지 않는다. 대신 **자바 → 코틀린으로 바뀌는 부분을 `diff` 코드블록(빨간 `-`/초록 `+`)으로** 보여주고, 그 변경된 부분만 구체적으로 설명한다. 여러 단계(예: 단순 변환 → 프로퍼티 접근 → 코틀린스럽게)로 나뉘는 강이면 각 단계 전환도 diff로 표현한다.
2. **새로 추가된 부분만** 다음 방식으로 설명한다:
   - 구체적인 예시 입력값을 정해서, 한 줄씩 실행되며 변수가 어떤 값이 되는지 순서대로 손으로 트레이스한다 (디버거는 사용하지 않음).
   - 자바 경험과 비교해서 설명한다 (이 강의 자체가 자바→코틀린 변환이므로 자연스럽게 매 강 적용됨).
3. **구조적으로 새로운 개념**(계층 분리, 데이터 흐름, 아키텍처 변경 등)이 나오면 정적 아키텍처 다이어그램(컨테이너 박스 + 화살표, 인터랙티브 아님)으로 시각화한다. 단순 문법 설명에는 무리해서 붙이지 않는다.
   - 시각화 도구로 먼저 보여주고, 해당 단계 로그 파일에도 **Mermaid 다이어그램으로 변환해서** 기록한다.
4. **(10강부터)** 사용자가 승인하면, 에이전트가 곧바로 다음을 전부 이어서 수행한다 — 중간에 다시 물어보지 않음:
   - 제시한 코드/변경사항을 실제 파일에 직접 반영 (자바 원본 삭제 + `.kt` 파일 생성/수정 등)
   - 빌드/테스트 실행해 검증
   - 현재 단계 학습 로그를 완료 상태로 갱신 (일반화 가능한 규칙이 있으면 `RULES.md`에도 추가)
   - `git add -A && git commit && git push` 실행 (매번 확인 불필요 — 사전 승인됨), 커밋 메시지는 한글로 `"N강: 내용 요약"` 형식
   - **바로 다음 강의 코드/설명까지 이어서 제시** (사용자가 별도로 "다음"이라고 말할 필요 없음)
5. (1~9강 방식, 참고용) 사용자가 직접 타이핑하던 시절엔 다 입력 후 "N강 됐어"로 알리면 에이전트가 위 검증/로그/커밋을 수행했다.

## 질문·트러블슈팅 기록 규칙

- 사용자가 **"질문N. ~~~"** 형식으로 물으면 현재 단계 로그 파일에 기록한다.
- 그 형식이 아니어도 질문이 troubleshooting인지 개념 질문인지 에이전트가 판단해 자동으로 기록한다.
- 기록 형식: `### 질문N` → **Q.**(질문 요약) / **A.**(답변 요약) / 있다면 **트러블슈팅**(증상/원인/해결).
- 특별한 질문 형식을 강요하지 않는다 — 평소처럼 물어보면 된다.

## 학습 로그 구조

- 위치: `docs/learning-log/step-NN.md` (NN은 강 번호 2자리), 템플릿은 `docs/learning-log/TEMPLATE.md`
- 섹션: 요구사항 요약 → 아키텍처 다이어그램(해당 시) → 질문 로그
- 로그 파일은 별도 커밋이 아니라 **해당 단계 코드와 같은 커밋에 포함**

## 환경 문제 대응 원칙

- 환경/설정 관련 버그(인코딩, 버전 호환 등)를 고치면 안내만 하고 끝내지 않는다. **설정 파일 자체를 고쳐서 커밋**해 다음에 반복 안 해도 되게 한다.
- 단, 이걸 별도 인프라 문서·권한 제한 시스템으로 과하게 확장하지 않는다 — **학습 프로젝트는 가볍게, git 자동화 + 학습 로그 기록에만 집중.** (필요하면 사용자가 명시적으로 요청할 것)

### 팀 프로젝트 적용용 변환 규칙 문서 (6강부터 추가)

학습 로그(`step-NN.md`)와 별개로, **다른 프로젝트(팀원의 자바 기반 프로젝트)에도 그대로 적용 가능한 일반화된 변환 패턴**만 뽑아서 `docs/conversion-rules/RULES.md`에 누적 기록한다.

- 대상: 6강부터 나온 규칙 (1~5강은 개념/환경 설정 위주라 일반화할 코드 규칙이 적음). 단, `build.gradle.kts`에 `kotlin("jvm")` 추가처럼 **build-config 성격의 규칙도 포함**한다.
- 형식: 규칙 ID(`R-NNN`) + 제목 + 카테고리(build-config/entrypoint/entity/dto 등) + 도입 강(링크) + Before/After 코드 + 적용조건 + 주의사항
- 스코프 원칙: 이 문서에는 **실제로 검증된(빌드/테스트 통과한) 규칙만** 넣는다.
- **자바 원본 파일 처리 방식 (9강부터 확정)**: 도메인 클래스를 변환할 때는 자바 원본을 **삭제하고 같은 경로에 `.kt`로 대체**한다 (강사의 실제 커밋과 동일). 6강에서 시도했던 "자바 원본을 남기고 `sourceSets exclude`로 빌드에서만 제외" 방식은 파일이 많아질수록 `build.gradle.kts`에 exclude가 계속 늘어나 관리 부담이 커서 폐기했고, 6강도 소급 적용해 자바 원본을 삭제했다. 자바 원본이 궁금하면 git 히스토리에서 언제든 확인 가능하므로 디스크에 남겨둘 필요가 없다.
- 매 강이 "완료" 처리될 때, 그 강에서 일반화 가능한 규칙이 나왔다면 `RULES.md`에 규칙을 추가하고 목차 표도 갱신한다 (해당 강의 커밋에 함께 포함).
- 팀원 프로젝트에 적용할 에이전트 전달 방식은 아직 미정 (현재는 문서화만 진행). 추후 방식이 정해지면 이 메모에 갱신할 것.
